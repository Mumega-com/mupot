# Mupot v0.23.0 - Trusted Runtime

## Goal

Ship Mupot `v0.23.0 - Trusted Runtime` as a dependable self-hosted agent
control plane: an operator can deploy it, connect a real agent, grant limited
authority, run and approve work, verify the result in GitHub, and recover the
system without losing state.

This release is not a marketplace, economy, new-department, full SOS-retirement,
GCP portability, or autonomous-brain expansion release. Those remain explicitly
outside the `v0.23.0` gate.

## Non-Negotiable Gates

The release is not ready until all of these are true:

- A fresh operator deploys a pot to Cloudflare, finishes owner setup, and proves
  no manual database edits were needed. The attached `fresh-install-check.json`
  must report `receipt_type:"mupot-fresh-install/v1"` and `status:"pass"` in
  #282.
- #274 has a real Hadi-host exported receipt bundle attached or linked, including
  `manifest.json`, `cutover-gate.json`, `export-receipt.json`, and
  `manifest-check.json` plus every receipt artifact named by the manifest. The
  final #284 checker must rerun the read-only fleet manifest verifier against
  that complete copied directory and report `status:"pass"`.
- One real product task moves from creation to real-agent execution, approval,
  completion, and audit visibility. The attached `work-lifecycle-check.json`
  must report `receipt_type:"mupot-work-lifecycle/v1"` and `status:"pass"` in
  #283.
- #150 proves one complete board to task to agent to GitHub PR cycle. The
  attached `external-pr-cycle-check.json` must report
  `receipt_type:"mupot-external-pr-cycle/v1"` and `status:"pass"`.
- #151 proves the live GitHub App has least-privilege permissions only:
  Contents R/W, Issues R/W, Pull requests R/W, Metadata R, Organization
  Projects R, Workflows disabled, and no extra org-admin permissions. The
  attached `github-app-permissions-check.json` must report
  `receipt_type:"mupot-github-app-permissions/v1"` and `status:"pass"`. The
  aggregate release-readiness bundle must also include `github-app.json` and
  `github-installation.json` with the exact permission set and hashes matching
  the passing permission receipt. The authenticated App-JWT exporter reads
  `GET /app` plus `GET /app/installations/{installation_id}`. When the App key

  `gh api apps/{slug}` plus the organization-owner installation endpoint;
  both paths emit the same redacted schemas and strict checker receipt.
- #277 remains closed: the fleet-control adversarial gate's runtime-control
  findings are fixed and covered by regression tests. #151 and #319 provide
  the corresponding GitHub least-privilege and task-mirror binding controls.
  This is the P0/P1 boundary for the Mupot Trusted Runtime scope. The distinct
  SOS/MCP hardening backlog (#182, #184, #185, #188) remains open and deferred
  under the release's explicit no-SOS-retirement constraint.
- A staging rehearsal proves upgrade, backup, restore, rollback, Queue/DLQ
  behavior, and failure reporting. The attached
  `staging-recovery-check.json` must report
  `receipt_type:"mupot-staging-recovery-rehearsal/v1"` and `status:"pass"`

- #323's tagged and deployed release candidate is retained in the aggregate

- `v0.23.0-rc.1` completes a seven-day production soak with no lost work,
  duplicated effects, unauthorized actions, or critical failures. Each daily

  `receipt_type:"mupot-production-soak/v1"` and `status:"pass"` in #280.

  milestone, the Git tag, and the GitHub Release all name the same final

## Release Candidate Gates

Before starting the `v0.23.0-rc.1` production soak:

```bash
npm audit --audit-level=high
npm run typecheck
npm test
node --test fleet-runtime/*.test.mjs
bash scripts/ci-local-evidence.sh
npm run receipt:fresh-install:plan -- --pot <pot> --base-url <pot-url> --operator <operator>
npm run receipt:github-app-permissions:plan -- --app <app-slug>
npm run receipt:work-lifecycle:plan -- --pot <pot> --base-url <pot-url> --agent <agent> --task-id <task-id>
npm run receipt:external-pr-cycle:plan -- --repo <owner/repo> --agent <agent> --task-id <task-id> --issue-url <issue-url> --pr-url <pr-url>
npm run receipt:staging-recovery:plan -- --pot <staging-pot> --base-url <staging-url>
npm run receipt:production-soak:plan -- --pot <pot> --base-url <pot-url> --rc-version v0.23.0-rc.1 --agent <agent>
npm run receipt:release-candidate:plan -- --version v0.23.0-rc.1 --repo <owner/repo>
npm run receipt:release-integrity:plan -- --version v0.23.0 --repo <owner/repo>
npm run receipt:release-readiness:plan -- --version v0.23.0 --repo <owner/repo> --checks-pr <release-pr-number>
npx wrangler deploy --dry-run --config wrangler.example.toml
```

GitHub Actions must also be green for:

- `build`
- `plugin`
- `no-secrets`
- `local-evidence`
- CodeQL analyses

The final #284 aggregate bundle must export both `github-pr.json` from
`gh pr view <release-pr-number>` and `github-checks.json` from
`gh pr checks <release-pr-number>`. The readiness check must be run with the
same `--checks-pr <release-pr-number>` value so the receipt can prove the CI
evidence came from the intended release-candidate PR.
