## Topology classification

- **A = headless-CLI-on-customer-infra** (mupot mints a token, runs the binary on the customer's own compute, wires `.mcp.json`/`config.toml`/`mcp.json` to mupot's MCP as a remote server): **Claude Code CLI, Codex CLI, Cursor CLI**. All three confirmed headless flags + confirmed remote-MCP-client support — this is the strongest-fit topology and works for all three today.
- **B = interactive-desktop/IDE-MCP-client** (human drives it; mupot's only lever is being an MCP server the app can add, no dispatch/monitor): **Claude Desktop app**. Codex's ChatGPT desktop app is closer to C (webhook-internal, GitHub-triggered) than B since there's no local-MCP-client story documented for it.
- **C = vendor-hosted-cloud-agent-via-API** (mupot calls a vendor REST API to launch + get results back): **Cursor Background Agents** (full launch+poll+webhook, best-documented of the three) and **Claude Managed Agents** (full launch+poll, no webhook). **Codex Cloud/ChatGPT desktop is NOT in this category** — no public launch/poll API exists (confirmed absent from docs + an open, unresolved OpenAI GH issue asking for exactly this contract).

## Flags — where the "governable by mupot" assumption is false or blocked

1. **Claude Desktop app is not a dispatch target.** No CLI, no API, no headless mode. mupot can at best be *listed* as a Custom Connector a human adds manually in Settings — mupot cannot launch, monitor, or govern tasks running inside it. If the spec lists Desktop as a governed harness row, that row should say "MCP-server-only, human-in-the-loop, no programmatic control" rather than implying parity with the CLI harnesses.
2. **Codex Cloud / ChatGPT desktop app has no public task API.** Confirmed by absence in `learn.chatgpt.com/docs/cloud` and by OpenAI's own open GitHub issue #24777 explicitly asking for "a stable public CLI/API contract" for Codex Cloud — i.e. OpenAI itself acknowledges the gap. The only governable OpenAI surface today is **Codex CLI run headlessly on infra mupot controls** (topology A) or `openai/codex-action` inside a customer's own CI (still A, just CI-flavored). Do not spec a "Codex cloud" adapter row that assumes launch+webhook — it would be vaporware against current docs.
3. **Cursor Background Agents API is beta, not GA** — "APIs may change before general availability." Base path is confirmed `api.cursor.com/v1` from the official Endpoints doc, but third-party/forum sources reference `v0` in places — verify live against the OpenAPI spec/dashboard before hardcoding a version in the adapter. No published hard rate-limit numbers. A forum thread reports agents spawned via the API cannot post PR comments/reviews even with correct GitHub App permissions — a real gap if the mupot adapter's "done" signal is meant to be a PR comment rather than the webhook/poll status.
4. **Claude Managed Agents has no webhook** — poll or SSE-stream only. If the spec's `runtime-adapter/v1` contract assumes a webhook completion callback uniformly across all C-topology harnesses, that assumption holds for Cursor but not for CMA; CMA needs a poll/SSE listener in the adapter instead.
5. **MCP connectivity for Cursor Background Agents specifically (not the CLI) is unconfirmed** — plausible per third-party writeups, no official doc pinned the exact config field. Verify before the spec asserts it.
